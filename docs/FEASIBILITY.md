# 可行性與難點分析 (Feasibility & Challenges)

對象：擬人化盆栽幽默日記系統。架構與規格見 [../CLAUDE.md](../CLAUDE.md)、[SPEC.md](SPEC.md)。

> **範圍註記（2026-06）：** 本專題**不做功耗實驗與 HCI 人體實驗**，聚焦 L2→L3→App 的軟體呈現；硬體一律 **USB 供電**。

---

## 1. 總體可行性結論

**整體可行性：高。** 三層皆建立在成熟、現成 (COTS) 技術上，沒有需要突破性研究的環節；本專題的創新在於**整合**與**互動設計**，而非發明新硬體或新演算法。

| 層 | 技術成熟度 | 可行性 | 主要不確定性 |
|---|---|---|---|
| L1 邊緣感測 | 高（ESP32 + 常見感測器） | ✅ 高 | ADC 校準、接線（USB 供電，無功耗顧慮） |
| L2 中介處理 | 高（Python/Pandas） | ✅ 高 | 觸發閾值需以真實資料調校 |
| L3 認知生成 | 高（LLM API） | ✅ 高 | 幻覺控制、語氣穩定、成本 |
| 呈現 / 整合 | 高 | ✅ 高 | 端到端時序與可靠度 |

**一句話：** 「會運作的 demo」風險低（目前 L2→L3→App 已跑通）；主要不確定性在**硬體到貨**與 **LLM 內容品質**，皆有對策。

---

## 2. 難點逐項分析

### 難點 1：感測雜訊與校準 (Sensor Noise) — ★★★☆☆
- **問題：** ESP32 ADC 有已知**非線性與雜訊**；電容式土壤感測器會隨溫度、土質、插入深度漂移。
- **真正的坑：**
  - 不同土壤的「乾/濕」ADC 值差異大，校準必須對應實際土壤。
  - 中位數濾波能去突波，但**系統性偏差**要靠校準曲線，兩者不可互相取代。
- **降風險：** L2 集中做校準（`calibration.json` 兩點校準）；硬體到貨後實測「全乾（懸空）/全濕（泡水）」兩點 raw 填入即可，**L2/L3 程式不動**。

### 難點 2：語義代溝 / LLM 幻覺 (Semantic Gap) — ★★★★☆
- **問題：** LLM 不擅長處理原始數字、且可能**虛構事件**或破壞人格（冒出「我是 AI…」）。
- **本專題的好設計：** 由 L2 先把數字轉成**狀態標籤 + 統計摘要**，LLM 只依標籤寫作 → 大幅降低算錯/幻覺。這是整個架構最聰明的一步。
- **仍存在的坑：**
  - 即使給了標籤，LLM 仍可能「加戲」捏造未提供的細節（例如自稱「昨天有蝴蝶來」）。需在 prompt 明令禁止，並抽樣人工檢核。
  - **語氣一致性**：同一人格在不同狀態下要像同一株植物，需 few-shot 範例與固定 system prompt。
  - **中文幽默品質**：幽默很主觀，模型與 prompt 需反覆試。
  - **延遲/離線**：雲端 API 偶發逾時，需重試與快取。
- **降風險：** 嚴格生成約束 + few-shot + 「只能依據提供資料」硬性規則 + 輸出後處理（過濾違規字串）。已在 `generator.postprocess` 與 `persona.md` 落實。

### 難點 3：成本 (LLM API) — ★★☆☆☆
- **問題：** 每日多次呼叫 LLM 會累積 API 成本。
- **降風險：**
  - 只在**狀態變化**或**每日一次**才呼叫 LLM（目前 seed / demo 已這樣做）。
  - 可改用本機 **Ollama**（免費）或有免費額度的 **Gemini**。
  - 小模型（gpt-4o-mini / gemini-flash）單次成本極低（一篇日記約 $0.0002）。

### 難點 4：系統整合與可靠度 — ★★★☆☆
- **問題：** 多層、（接硬體時）跨裝置、非同步；任何一段斷掉，使用者就看不到日記。
- **坑：** 時間戳對齊、MQTT 斷線重連、資料缺漏時的降級策略、各模組由不同人開發的介面落差。
- **降風險：** 介面合約**先凍結**，並寫**契約測試**；各層可用假資料獨立開發（已落實：日記先 seed 進 DB，網頁只讀 DB）。

## 3. 風險矩陣 (Risk Matrix)

| 風險 | 可能性 | 衝擊 | 對策 |
|---|---|---|---|
| 硬體到貨延遲 | 中 | 中 | 以模擬資料平行開發；MVP 僅需土壤感測器；備選 DHT22 替 BME280 |
| ADC 校準不準 | 中 | 中 | 多點校準 + 中位數濾波；固定插入深度 |
| LLM 幻覺 / 破格 | 中 | 中 | 嚴格約束 + few-shot + 輸出過濾 + 人工抽檢 |
| LLM 額度 / 速率 (429) | 中 | 低 | 限頻生成、自動重試、可換供應商 / 本機 Ollama |
| API 成本超支 | 低 | 低 | 限頻生成、設上限、用小模型 |
| 介面不一致導致整合失敗 | 中 | 高 | 凍結合約 + 契約測試 + 假資料平行開發 |

## 4. 給「下週二展示」的務實建議

1. **必做（已完成）：** 土壤濕度（模擬或實測）→ Pandas 出狀態 → LLM 生成日記 → 網頁雙畫面呈現。
2. **可模擬：** 溫濕度 / 光照（BME280 / LDR 未到貨前用模擬值，封包標 `sim:true`）。
3. **展示腳本：** 故意讓土壤「變乾」（或注入乾燥資料）→ 即時看到植物「喊渴」的幽默日記，對比同一時刻的傳統儀表板 → 直擊「情感斷層」訴求。

這樣即使硬體未到位，也能完整傳達研究主張，且每一層都有真材實料。

## 5. 教學注意事項 (給授課者)

學生最容易卡關 / 最常犯錯之處：
- **校準 vs 濾波：** 兩者目的不同，常被學生混為一談。
- **LLM 數字能力：** 別讓學生把原始陣列丟給 LLM；強調「先轉標籤」的設計。
- **介面先行：** 不先凍結 JSON / 標籤就各寫各的，整合日必爆。建議第一堂就一起定合約。
- **思考模型的 token：** 用 Gemini 2.5 等「思考」模型時，輸出 token 上限要給足，否則可見文字會被截斷（本專案 `LLM_MAX_TOKENS` 預設 1024）。
- **可分工性：** 合約凍結後，C / D 用假資料即可開工，不必等硬體。
