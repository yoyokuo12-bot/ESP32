# 可行性與難點分析 (Feasibility & Challenges)

對象：擬人化盆栽幽默日記系統。架構與規格見 [../CLAUDE.md](../CLAUDE.md)、[SPEC.md](SPEC.md)。

---

## 1. 總體可行性結論

**整體可行性：高。** 三層皆建立在成熟、現成 (COTS) 技術上，沒有需要突破性研究的環節；本專題的創新在於**整合**與**互動設計**，而非發明新硬體或新演算法。

| 層 | 技術成熟度 | 可行性 | 主要不確定性 |
|---|---|---|---|
| L1 邊緣感測 | 高（ESP32 + 常見感測器） | ✅ 高 | 省電續航的「實測值」、ADC 校準 |
| L2 中介處理 | 高（Python/Pandas） | ✅ 高 | 觸發閾值需以真實資料調校 |
| L3 認知生成 | 高（LLM API） | ✅ 高 | 幻覺控制、語氣穩定、成本 |
| 呈現/整合 | 高 | ✅ 高 | 端到端時序與可靠度 |
| HCI 評估 | 中（流程繁瑣） | ⚠️ 中 | 招募 30 人 × 30 天、倫理、統計效力 |

**一句話：** 做出「會運作的 demo」風險低；做到「簡報宣稱的量化結果（14.5 個月續航、嚴謹 HCI 結論）」風險中等，主要卡在**時間**與**實驗嚴謹度**。

## 2. 難點逐項分析（對應簡報「三大鴻溝」+ 實務）

### 難點 1：硬體功耗 (Hardware Power) — ★★★☆☆
- **問題：** 若 Wi-Fi 常時連線（~115 mA）電池僅約 21.7 小時；要達數月續航必須靠深度睡眠。
- **真正的坑：**
  - **Wi-Fi 重連才是耗電大宗**，不是睡眠電流。每次喚醒的 DHCP/關聯握手可能耗時 1–3 秒、瞬間電流數百 mA；省電成敗在這裡。
  - 開發板上的 LED、USB-UART 晶片、穩壓器靜態電流會「吃掉」µA 級的睡眠優勢；**裸 ESP32 模組**才能達標。
  - 18650 自放電、低溫掉電壓、TP4056 靜態消耗都會讓「14.5 個月」理論值縮水。
- **降風險：** 先量測真實單次喚醒能耗 → 反推續航，別只引用理論；展示時用「數天實測 + 外推」即可，不必真的等數月。
- **教學提醒：** 學生常忽略 Wi-Fi 關聯耗電、用開發板量睡眠電流得到偏高數值。

### 難點 2：感測雜訊與校準 (Sensor Noise) — ★★★☆☆
- **問題：** ESP32 ADC 有已知**非線性與雜訊**；電容式土壤感測器會隨溫度、土質、插入深度漂移。
- **真正的坑：**
  - 不同土壤的「乾/濕」ADC 值差異大，校準必須對應實際土壤（簡報用壤質粉砂土做 5 點校準是對的做法）。
  - 中位數濾波能去突波，但**系統性偏差**要靠校準曲線，兩者不可互相取代。
- **降風險：** L2 集中做校準（見 CLAUDE.md §9）；先用 `calibration.json` 兩點/多點對照，之後用 §SPEC F-E2 的砝重法精修。
- **可達性：** CV 6.5%–10.3% 屬合理範圍，可達成。

### 難點 3：語義代溝 / LLM 幻覺 (Semantic Gap) — ★★★★☆
- **問題：** LLM 不擅長處理原始數字、且可能**虛構事件**或破壞人格（冒出「我是 AI…」）。
- **本專題的好設計：** 由 L2 先把數字轉成**狀態標籤 + 統計摘要**，LLM 只依標籤寫作 → 大幅降低算錯/幻覺。這是整個架構最聰明的一步。
- **仍存在的坑：**
  - 即使給了標籤，LLM 仍可能「加戲」捏造未提供的細節（例如自稱「昨天有蝴蝶來」）。需在 prompt 明令禁止，並抽樣人工檢核。
  - **語氣一致性**：同一人格在不同狀態下要像同一株植物，需 few-shot 範例與固定 system prompt。
  - **中文幽默品質**：幽默很主觀，模型與 prompt 需反覆試。
  - **延遲/離線**：雲端 API 偶發逾時，需重試與快取。
- **降風險：** 嚴格生成約束 + few-shot + 「只能依據提供資料」硬性規則 + 輸出後處理（過濾違規字串）。

### 難點 4：成本 (LLM API) — ★★☆☆☆
- **問題：** 30 盆 × 30 天 × 每日多次喚醒 → API 呼叫量與費用累積。
- **降風險：**
  - 不必「每筆 telemetry 都生成」——只在**狀態變化**或**每日一次**才呼叫 LLM。
  - 評估期可改用本機模型（Ollama）或較小模型控成本。
  - 先估算：呼叫頻率 × token × 單價，設每月上限。

### 難點 5：系統整合與可靠度 — ★★★☆☆
- **問題：** 三層、跨裝置、非同步；任何一段斷掉，使用者就看不到日記。
- **坑：** 時間戳對齊、MQTT 斷線重連、資料缺漏時的降級策略、各模組由不同人開發的介面落差。
- **降風險：** §2 介面合約**先凍結**，並寫**契約測試**；各層可用假資料獨立開發。先求單盆端到端跑通，再擴充。

### 難點 6：HCI 實驗嚴謹度 — ★★★★☆（時間與方法風險最高）
- **問題：** 招募 ~30 人、為期 30 天、A/B 對照，是本專題**最耗時、最易失分**的部分。
- **坑：**
  - 招募與留存（30 天每日回填問卷，流失率高）。
  - **研究倫理**：受試者同意書、個資（若做問卷/可能需 IRB 或課程倫理規範）。
  - 變因控制：兩組植物初始狀態、環境、品種需一致。
  - **統計效力**：n=30 分兩組，每組 ~15，效力有限，結論要保守陳述。
- **降風險：** 若時間不足，可縮為較短週期 + 質性訪談 + 小樣本，並在報告誠實標註限制；先確認課程是否要求/允許正式人體研究。

## 3. 風險矩陣 (Risk Matrix)

| 風險 | 可能性 | 衝擊 | 對策 |
|---|---|---|---|
| 硬體到貨延遲 | 中 | 高 | C/D 先用假資料平行開發；備選 DHT22 替 BME280 |
| 續航實測遠低於理論 | 中 | 中 | 改報「實測 + 外推」，誠實標示量測條件 |
| ADC 校準不準 | 中 | 中 | 多點校準 + 中位數濾波；固定插入深度 |
| LLM 幻覺/破格 | 中 | 中 | 嚴格約束 + few-shot + 輸出過濾 + 人工抽檢 |
| API 成本超支 | 低–中 | 中 | 限頻生成、設上限、可換本機模型 |
| HCI 招募/倫理卡關 | 高 | 中 | 縮小規模、改質性、先確認倫理要求 |
| 介面不一致導致整合失敗 | 中 | 高 | W0 凍結合約 + 契約測試 |

## 4. 給「下週二展示」的務實建議 (MVP 切法)

時間有限時，**先確保有一個會動的端到端故事**，其餘用模擬/預錄補：

1. **必做：** 單盆 + 土壤濕度一個感測器 + MQTT → Pandas 出 `CRITICAL_DROUGHT`/`STABLE` → LLM 生成一篇真實日記，畫面呈現。
2. **可模擬：** 溫濕度/光照、深度睡眠續航（用「實測數筆 + 外推」說明）、HCI 結果（用設計好的問卷 + 少量試填）。
3. **展示腳本：** 故意讓土壤變乾（或注入乾燥資料）→ 即時看到植物「喊渴」的幽默日記，對比同一時刻的傳統儀表板 → 直擊「情感斷層」訴求。

 這樣即使硬體/實驗未完整，也能完整傳達研究主張，且每一層都有真材實料。

## 5. 教學注意事項 (給授課者)

學生最容易卡關 / 最常犯錯之處，建議課堂先點出：
- **省電量測：** 用開發板（含 LED/UART）量睡眠電流會偏高；Wi-Fi 重連耗電被忽略。
- **校準 vs 濾波：** 兩者目的不同，常被學生混為一談。
- **LLM 數字能力：** 別讓學生把原始陣列丟給 LLM；強調「先轉標籤」的設計。
- **介面先行：** 不先凍結 JSON/標籤就各寫各的，整合日必爆。建議第一堂就一起定 §SPEC.md §2。
- **倫理與隱私：** 凡涉及人（HCI 問卷）就要談同意與個資，趁機帶研究倫理。
- **可分工性：** 合約凍結後，C/D 用假資料即可開工，不必等硬體 —— 這是讓全組同時動起來的關鍵。
